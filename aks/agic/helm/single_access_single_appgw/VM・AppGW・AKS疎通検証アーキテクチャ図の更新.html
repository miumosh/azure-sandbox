⚠️ 4. 制限事項

AKS 制限事項
  • AKS SKU: Standard 以上必須 (Free ティアでは AGIC 動作不可)
  • AGIC バージョン: v1.9.1 以上必須 (CNI Overlay サポート対応)
  • CNI Overlay 固有制限: VM Availability Sets および DCsv2-series VM は非対応
  • Kubernetes バージョン: LTS 専用パッチバージョンを使用する場合は Premium SKU 必須
Application Gateway 制限事項
  • SKU 制限: Standard_v2 または WAF_v2 のみサポート
  • フロントエンド IP: v2 はプライベート IP のみの構成 (Private Deployment) も可能。本検証ではパブリック IP ＋ プライベート IP の両方を設定
  • AppGW サブネット: Microsoft.Network/applicationGateways の Delegation (委任) が必須。サイズは /24 以下を推奨

Service Type に関する制約と検証結果

  • NodePort / ClusterIP 双方で疎通確認済み: 今回の検証では両方の Type で疎通が成功することを確認
  • AGIC v1.9.1 未満の場合: ServiceType NodePort が必須 (AppGW バックエンドプールには Node VNet IP : NodePort が登録される)
  • AGIC v1.9.1 以上 ＋ Route Table 関連付け済みの場合: ClusterIP でも疎通可能 (AppGW バックエンドに Pod IP が直接登録され、VNet ルーティング経由で到達)

必須 NSG ルール (Azure CNI Overlay)

Azure CNI Overlay においてカスタム NSG を使用する場合、以下の通信許可ルールの設定が必須。

優先度 ルール名 方向 送信元 宛先 ポート 目的
100 AllowNodeToNode Inbound aks-subnet aks-subnet * ノード間通信
110 AllowNodeToPod Inbound aks-subnet pod CIDR * Service から Pod への転送
120 AllowPodToPod Inbound pod CIDR pod CIDR * Pod 間通信・DNS (CoreDNS)
130 AllowAppGWToNodes Inbound appgw-subnet aks-subnet 30000-32767 AppGW から NodePort へのアクセス

🛣️ 5. 経路について — Ingress / Egress

🔽 Ingress 経路 (今回の検証対象)

経路: Client → AppGW (Private IP) → AKS Node (NodePort / Pod IP) → Pod

外部や社内網からクラスター内のサービスへ到達するための「受信トラフィック」の制御。本検証の主目的として動作確認済み。

🔼 Egress 経路 (今回未検証・参考情報)
  • デフォルト動作: AKS 既定の Egress は outboundType = loadBalancer。Pod から外部への通信は Node IP で SNAT 後、Standard LB 経由で送信。
  • ファイアウォール経由要件: FW 等を強制経由する場合は userDefinedRouting ＋ UDR 設定が必要。
  • WAF 利用時の注意: AppGW WAF_v2 利用時、AppGW サブネットに UDR の追加設定が必要となる想定。ただし誤った UDR は非対称ルーティングやヘルスプローブ異常の要因となるため、管理トラフィックのインターネット直接送信設計に注意が必要。

🛡️ 7. ネットワークポリシー比較

🟢 Cilium (Azure 推奨)
  • eBPF ベースのデータプレーン
  • L7 ポリシー / FQDN フィルタリング対応
  • 高パフォーマンス・高スケーラビリティ
  • Azure CNI Powered by Cilium で統合利用可能
  • Azure CNI Overlay 環境に完全対応
🔵 Azure NPM
  • iptables (Linux) / HNS ACLPolicies (Windows) ベース
  • ⚠️ Windows ノード: 2026年9月30日にサポート終了予定
  • Linux スケール上限: 250ノード / 20,000ポッド
  • Kubernetes 標準ネットワークポリシーに準拠
🟠 Calico
  • Tigera 社が提供するオープンソースソリューション
  • Kubenet (Linux のみ) および Azure CNI 環境に対応
  • Kubernetes 標準ポリシーに準拠
  • ※独自の Calico 拡張機能は AKS サポート対象外

🔑 8. 権限構成

8-1. Identity 全体像

⚙️ System-Assigned MI
(AKS Control Plane)
  • 用途: AKS が Azure API (VMSS, NIC, LB 等) を操作
  • ライフサイクル: AKS クラスターと同期（削除時自動削除）
  • 付与ロール: Network Contributor @ aks-subnet
👤 User-Assigned MI
(AGIC: agic-green-agic-identity)
  • 用途: AGIC Pod が Azure API (AppGW 等) を操作
  • ライフサイクル: クラスターとは独立（明示的な削除が必要）
  • 認証連携: Workload Identity / OIDC を利用
🔗 Federated Identity Credential (OIDC 連携の要)
  • issuer: [AKS OIDC Issuer URL]
  • subject: system:serviceaccount:kube-system:{helm-release-name}-sa-ingress-azure

⚠️ Helm の release name が変更されると ServiceAccount 名 (subject) も変わるため不一致エラーの原因となる。

8-2. Azure RBAC ロール割り当て一覧

AGIC UAI → Reader @ agic-green-rg  (RG 全体の読み取り)
AGIC UAI → Contributor @ AppGW  (AppGW 設定の読み書き)
AGIC UAI → Network Contributor @ appgw-subnet  (Route Table 関連付け)
AGIC UAI → ★ Network Contributor @ MC_ RG  重要: Route Table 読み取り（見落としやすい）
AKS CP MI → Network Contributor @ aks-subnet  (ノード NIC の接続)
⚠️ Helm デプロイにおけるロール割り当ての手動適用（推測含む）
AKS アドオン方式とは異なり、Helm を用いた AGIC インストールでは上記の Azure RBAC ロール割り当ては自動化されず、Terraform または Azure CLI を用いた手動設定が必要となる。特に自動作成される MC_ リソースグループへの Network Contributor 権限付与は見落としやすいため厳重な注意が必要。
🔍 未検証: ACR Pull 権限について
コンテナイメージを Azure Container Registry (ACR) からプルする場合、AKS ノードプールの Kubelet Managed Identity に対して AcrPull ロールの付与が必要。本検証では ACR を使用していないため実機確認は未実施だが、実運用時には別途設定が必須となる。

8-3. なぜ AGIC だけ Workload Identity (OIDC) が必要か

ポイント： AGIC Pod はクラスター内で唯一 Azure ARM API を直接呼び出すコンポーネントです。
K8s Pod は通常 Azure Managed Identity を直接利用できないため、 Workload Identity (OIDC) を用いて K8s SA トークンを Azure AD トークンに変換します。
コンポーネント 実行場所 認証方式 OIDC 使用 Azure ARM 操作
AKS Control Plane Azure マネージドサービス System-Assigned MI
(AKS 作成時に自動生成)		 ❌ 不要 VMSS / NIC / LB 操作
Kubelet (ノード) Azure VM (VMSS) Kubelet Managed Identity ❌ 不要 ACR Pull / ノード操作
アプリ Pod (NodePort/ClusterIP) K8s Pod 不要 (Azure API 非呼び出し) ❌ 不要 なし
🤖 AGIC Pod ← 唯一 K8s Pod User-Assigned MI
+ Workload Identity (OIDC) 		✅ 必要 AppGW 設定変更
Route Table 操作
なぜ Workload Identity が必要か：
K8s Pod は Azure VM ではないため、通常の Managed Identity (IMDS) を利用できません。 Workload Identity は OIDC トークン交換 の仕組みを使い、 「AKS OIDC Issuer が発行した K8s SA JWT」を Azure AD が信頼し Bearer Token を発行することで、 Pod が Azure ARM API を安全に呼び出せるようにします。
旧来の AAD Pod Identity（非推奨・廃止方向）に代わる現在の推奨方式です。

8-4. Workload Identity 認証フロー（詳細）

📐 セットアップフェーズ（Terraform / 事前設定）
①
AKS: OIDC Issuer 有効化 ＋ Workload Identity Webhook インストール
oidc_issuer_enabled = true workload_identity_enabled = true
→ AKS が OIDC Issuer エンドポイントを公開
→ Webhook が SA annotation を検知し Pod に認証情報を注入できるよう準備
②
Azure AD: Federated Identity Credential 登録
issuer = https://oidc.prod.aks.azure.com/<tenant>/<uuid>/
subject = system:serviceaccount:kube-system:agic-sa-ingress-azure
→ この OIDC Issuer が発行した、この SA のトークンを信頼する設定
⚠️ subject は Helm の release name に依存 — 変更時は再設定が必要
🚀 実行時フェーズ（AGIC Pod 起動時）
③
Workload Identity Webhook が SA アノテーションを検知
azure.workload.identity/client-id: <agic-identity-client-id>
AGIC Pod の ServiceAccount に上記 annotation が付いていることを Webhook が確認
④
環境変数 ＆ Projected Volume を AGIC Pod に注入
AZURE_CLIENT_ID = <UAI の client ID> AZURE_TENANT_ID = <テナント ID>
AZURE_FEDERATED_TOKEN_FILE = /var/run/secrets/azure/tokens/…
→ Projected ServiceAccount Token (audience: api://AzureADTokenExchange) が volume として提供される
🔐 Azure AD トークン交換フェーズ
⑤
AGIC SDK が Federated Token ファイルから K8s SA JWT を読み込み
JWT ペイロード例:
iss: https://oidc.prod.aks.azure.com/… sub: system:serviceaccount:kube-system:agic-sa-ingress-azure aud: ["api://AzureADTokenExchange"]
⑥
Azure AD が Federated Identity Credential と照合
issuer 一致確認 → ✅
subject 一致確認 → ✅
② で登録した設定と完全一致した場合のみ認証成功
✅ 認証完了 → Azure ARM API 呼び出し可能
⑦
Azure AD が User-Assigned MI の Bearer Token を発行
AGIC は Azure Resource Manager API を呼び出し可能な状態に
→ AppGW のバックエンドプール / ルーティングルール / ヘルスプローブ の CRUD
→ MC_RG の Route Table 読み取り ＋ appgw-subnet への関連付け

8-5. CNI Overlay Route Table フロー

Step 1

AKS が MC_ リソースグループ内に Route Table を自動作成。
例: 192.168.0.0/24 → Node 0

Step 2

AGIC が自動作成された Route Table を AppGW サブネットに関連付け。
(※ Network Contributor 権限が必要)

Step 3

AppGW から VNet 外の Pod IP (192.168.x.x) へ VNet ルーティング経由で直接パケット到達が可能になる。

🚫 9. 権限不足エラーパターン

AUTH AADSTS700213
Federated Identity Credential の subject (SA 名) 不一致エラー。Helm のリリース名変更が主な原因。
AUTH AADSTS70011
identityClientID の設定誤り。Helm values への正しい Client ID の渡し忘れ。
権限 403 routeTables/read
MC_ RG に対する Network Contributor ロールの付与漏れ。
権限 Unhealthy routeTables/join/action
appgw-subnet に対する Network Contributor ロールの付与漏れ。関連付けに失敗。
起動 context deadline exceeded
権限や認証エラー起因による AGIC Pod の CrashLoopBackOff。
設定 K8sVersionNotSupported
Standard SKU における LTS 専用パッチバージョンの指定エラー。
